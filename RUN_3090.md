# Running the Full Paper Experiment on an RTX 3090

This guide covers the **full 3-dataset experiment** (NQ-Open, HotpotQA, RGB √ó
C0/C1/C2/C3 √ó K‚àà{1,3,5}) on a single **RTX 3090 (24 GB VRAM, Ampere sm_86)**.

> Note: the RTX 3090 has **24 GB** of VRAM (not 30 GB). All estimates below assume 24 GB.

---

## TL;DR commands

```bash
export SCRATCH_DIR=/mnt/nvme/turborag_quant     # MUST be fast local NVMe (I/O bound)
export HF_HOME=/mnt/nvme/hf_cache

# 1) Small validation run (~30‚Äì45 min, ~12 GB disk)
python src/run_experiment.py --config configs/full_experiment.yaml \
    --wiki_pages 1000 --num_nq_examples 50 --num_hotpot_examples 50 --num_rgb_examples 50

# 2) Paper run (~6‚Äì9 h, see storage table)
python src/run_experiment.py --config configs/full_experiment.yaml \
    --wiki_pages 10000 --num_nq_examples 200 --num_hotpot_examples 200 --num_rgb_examples 200

# 3) Large stress run (needs ~250 GB disk)
python src/run_experiment.py --config configs/full_experiment.yaml \
    --wiki_pages 25000 --num_nq_examples 200 --num_hotpot_examples 200 --num_rgb_examples 200
```

Every value is a CLI flag ‚Äî **no source edits are ever required**. The resolved
run config (including `wiki_pages`) is written to `$SCRATCH_DIR/results/config.json`
and echoed into every `results.csv` / `summary.json` row and the `logs_*.txt`.

## The binding constraint is DISK, not VRAM

KV-cache geometry for `Qwen2.5-3B-Instruct` (36 layers, 2 KV heads, head_dim 128):

| Precision | Bytes / token | Relative |
|---|---|---|
| FP16 | 36.0 KB | 1.00√ó |
| INT8 | 19.1 KB | 0.53√ó |
| INT4 |  9.6 KB | 0.27√ó |
| **All three (C1+C2+C3)** | **64.7 KB** | ‚Äî |

Running C0‚ÄìC3 requires **all three** precision files per chunk.

### Disk estimate by `--wiki_pages` (all three precisions)

| `--wiki_pages` | Wiki KV cache | + HotpotQA (~18 GB) + RGB (~50‚Äì100 GB) | Total to provision |
|---|---|---|---|
| 1,000  |   9 GB | + ~70 GB | **~80 GB** |
| 5,000  |  46 GB | + ~70 GB | **~115 GB** |
| 10,000 |  92 GB | + ~70 GB | **~160‚Äì210 GB** |
| 15,000 | 138 GB | + ~70 GB | **~210‚Äì260 GB** |
| 25,000 | 230 GB | + ~70 GB | **~300‚Äì350 GB** |

> RGB documents are long web snippets (5 positive + 15 negative per question);
> their KV caches dominate after wiki. If disk is tight, lower `--num_rgb_examples`
> or `--wiki_pages` first. The KV caches are **intermediate** ‚Äî delete
> `$SCRATCH_DIR/chunk_kvcache` after the eval+analyze stages finish.

**Storage levers**
- Build only the precisions you need: `--conditions C0 C1` ‚Üí FP16 only (‚âà0.54√ó disk).
- The `--precisions` filter is applied automatically from `--conditions`.
- Put `kvcache_dir` on **local NVMe**. The eval stage re-reads chunk caches once
  per (example √ó K √ó condition), so network/HDD storage will dominate wall time.

## VRAM budget (peak ‚âà 10 GB of 24 GB ‚Äî comfortable)

| Component | VRAM |
|---|---|
| Qwen2.5-3B (float16) | ~6.2 GB |
| HHEM-2.1-Open (flan-t5-base) | ~1.0 GB |
| DeBERTa-v3-large NLI | ~1.7 GB |
| bge-small embedder | ~0.4 GB |
| Stitched KV cache @ K=5 (~2.6k tok) | ~0.1 GB |
| **Peak** | **~10 GB** |

No OOM risk. You have headroom to raise scorer batch sizes:
`evaluation.hhem_batch_size` / `nli_batch_size` ‚Üí **32‚Äì64** (set in
`configs/full_experiment.yaml`).

float16 is intentional: the stored caches are float16 and the quantization
fp16 path casts to float16, so the model must match. The 3090 *supports*
bfloat16, but switching would require changing `kv_quantization.py` too ‚Äî not
worth the numerical inconsistency.

## Runtime estimate (`--wiki_pages 10000`, 200 examples each)

| Stage | Work | Est. time |
|---|---|---|
| build | forward + quantize + save ~15k chunks, build index | ~1.5‚Äì2.5 h |
| eval  | 3 ds √ó 3 K √ó 4 cond √ó 200 = 7,200 generations + HHEM/NLI | ~4‚Äì6 h |
| calib | HHEM vs NLI correlation on 50 | ~3 min |
| analyze + tables | CPU only | <1 min |
| **Total** | | **~6‚Äì9 h** |

The eager-attention re-rotation of the full key sequence each decode step is
O(N¬≤); it is the per-token cost and is intrinsic to the raw-key TurboRAG design.

## Bottlenecks & mitigations

1. **Disk I/O (primary)** ‚Äî eval re-loads each chunk's `.pt` per example/K/condition.
   ‚Üí Local NVMe; consider lowering `--wiki_pages`; delete caches when done.
2. **Disk capacity (primary)** ‚Äî see table above. Provision before the run.
3. **NQ-Open recall (research)** ‚Äî a random DPR subset has limited recall for NQ
   questions. NQ hallucination/EM are only meaningful once `--wiki_pages` is large
   (‚â•10k, ideally more). Watch `refusal_rate` for `nq_open` in the summary ‚Äî a high
   value means the corpus does not cover the questions (expected at small wiki sizes).
4. **HHEM/NLI 512-token window** ‚Äî with `faithfulness_mode: full_context`, long
   concatenated contexts at K=5 are truncated by the scorer. `per_chunk_max`
   (default) avoids this but is lenient and can mask H2. **Run both and report.**
